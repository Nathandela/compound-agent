# v0.2.2 Implementation Plan

## Overview

This document defines the parallel implementation strategy for v0.2.2 "Hardening & Quality Gates" release.

## Feature Summary

| ID | Feature | Schema Changes | File Touches | Complexity |
|----|---------|----------------|--------------|------------|
| eik | Age-based temporal validity | compactionLevel, compactedAt, lastRetrieved | types.ts, sqlite.ts, session.ts, cli.ts (stats, load-session) | Medium |
| qp9 | Count warning | NONE | session.ts, cli.ts (stats, load-session) | Low |
| tn3 | Citation field | citation?: {file, line?, commit?} | types.ts, jsonl.ts, sqlite.ts, cli.ts (learn, list) | Medium |
| mov | Manual invalidation | invalidatedAt, invalidationReason | types.ts, jsonl.ts, sqlite.ts, retrieval/*.ts, cli.ts (new commands) | High |

## Dependency Analysis

### Schema Changes (All Additive - No Conflicts)
- **eik**: `compactionLevel: 0|1|2`, `compactedAt?: string`, `lastRetrieved?: string`
- **tn3**: `citation?: { file: string; line?: number; commit?: string }`
- **mov**: `invalidatedAt?: string`, `invalidationReason?: string`
- **qp9**: No schema changes

### File Conflict Matrix

| File | eik | qp9 | tn3 | mov | Resolution |
|------|-----|-----|-----|-----|------------|
| types.ts | Yes | No | Yes | Yes | Additive fields, merge safe |
| sqlite.ts | Index | Query | Index | Index+Query | Merge with care |
| jsonl.ts | No | No | Write | Write | Additive, merge safe |
| session.ts | Warning | Warning | No | Exclusion | Sequential merge |
| cli.ts | stats, load-session | stats, load-session | learn, list | wrong, validate, list | Different commands |

### Parallelization Decision: ALL 4 CAN RUN IN PARALLEL

Rationale:
1. Schema changes are additive (different fields)
2. CLI changes are to different commands
3. Retrieval changes are orthogonal (warning vs exclusion)
4. Integration testing after merge catches conflicts

## Parallel Execution Strategy

### Phase 1: Define Invariants (Parallel - 4 subagents)
Each subagent runs `/invariant-designer` for its feature.

### Phase 2: Write Tests (Parallel - 4 subagents)
Each subagent runs `/test-first-enforcer` + `/property-test-generator`.
Output: Test files that fail (red phase).

### Phase 3: Implement (Parallel - 4 subagents)
Each subagent implements minimal code to pass tests.
Uses feature branches: `feature/v0.2.2-eik`, `feature/v0.2.2-qp9`, etc.

### Phase 4: Merge & Integration (Sequential)
1. Merge all feature branches to `feature/v0.2.2-hardening`
2. Resolve any conflicts (should be minimal)
3. Run full test suite
4. Run `/implementation-reviewer`

### Phase 5: Final Review
Each feature reviewed by `/implementation-reviewer`.
All must be APPROVED before release.

## Merge Strategy

1. Each feature works on its own branch
2. Before merging, ensure all tests pass
3. Merge in order: qp9 (simplest) → tn3 → eik → mov (most complex)
4. After each merge, run full test suite
5. Resolve conflicts preserving all functionality
